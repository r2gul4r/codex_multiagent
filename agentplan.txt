# Agent Plan

## Goal

- `main` 은 계획, 계약 고정, 작업 분배만 담당
- 실제 코드 수정은 `worker` 들만 담당
- `reviewer` 는 최종 read-only 검수만 담당

## Core Model

- `main` 에게 코드 수정 권한을 주지 않음
- `main` 은 구현 전에 공유 계약을 먼저 고정함
- 공유 계약에는 API payload, 상태 이름, 호출 순서, 테스트 전제, 파일별 수정 범위를 포함함
- 계약이 고정되기 전에는 `worker` 를 투입하지 않음
- 계약 고정 후 `worker` 를 2~4개까지 병렬로 투입 가능

## Task Size Gate

- 작업 크기 판단은 `하드 트리거 + 점수제` 로 결정함
- `main` 은 수정 전에 먼저 하드 트리거를 보고, 없으면 점수를 계산함
- 작은 작업으로 시작했더라도 도중에 하드 트리거가 발견되면 즉시 큰 작업 라우트로 승격함

## Hard Triggers

- 아래 중 하나라도 걸리면 작은 작업으로 보지 않음
- 공유 계약 변경
  - API payload
  - 상태 이름 또는 상태 전이
  - 이벤트 이름
  - 라우트 이름
  - env key
- shared 자산 변경
  - 공용 타입
  - 공용 util
  - 공용 컴포넌트
  - import path
  - schema
- 수정 범위가 2개 이상 레이어로 퍼짐
  - UI + 서버
  - UI + 테스트
  - 서버 + schema
- write set 을 자연스럽게 둘 이상으로 분리할 수 있음
- 회귀 위험이 중간 이상임
- 구현 후 별도 reviewer 관점이 꼭 필요함

## Scorecard

- 하드 트리거가 없을 때만 점수제를 사용함
- 각 항목 1점
- 수정 파일이 3개 이상
- 디렉터리가 2개 이상
- 새 파일 생성이 2개 이상
- 테스트 수정이 필요함
- 기존 코드 이해가 선행돼야 함
- 구현 전에 설계 결정이 하나 이상 필요함
- 수동 검증 단계가 2개 이상 필요함

## Route Selection

- `0~1점`
  - 작은 작업
  - `main` 이 직접 수정 가능
- `2~3점`
  - 경계 작업
  - 기본은 `main` 단독 진행
  - 필요하면 `explorer` 또는 `reviewer` 만 추가
- `4점 이상`
  - 큰 작업
  - `main planner-only + parallel workers + reviewer`
- 하드 트리거가 하나라도 있으면 점수와 무관하게 큰 작업 후보로 봄

## Upgrade Rule

- 작은 작업으로 시작했더라도 아래 상황이 나오면 즉시 planner-only 로 전환함
- shared 계약 변경이 필요해짐
- shared 자산 수정이 필요해짐
- write set 이 둘 이상으로 자연 분리됨
- 회귀 위험이 예상보다 커짐
- `main` 이 이미 수정 중이어도 이 조건이 생기면 손을 멈추고 재분배함

## Main Decision Flow

1. 요구사항을 한 줄 acceptance criteria 로 압축
2. 바뀔 파일, 레이어, shared 계약 후보를 빠르게 스캔
3. 하드 트리거 확인
4. 하드 트리거가 없으면 점수 계산
5. `Route A`, `Route B`, `Route C` 중 하나 선택

## Execution Diagram

```text
사용자 프롬프트
-> bridge `/app/run`
-> 현재 턴용 delegation authorization prefix 자동 주입
-> main 이 `agentplan` 기준으로 triage
-> Route A / Route B / Route C 결정
-> spawn gate 확인
-> Route A/B 는 main 중심 진행
-> Route C 는 worker / reviewer 호출
-> reviewer 검수
-> main 종료 판단
```

## Spawn Gate Resolution

- `main` 의 triage 와 실제 sub-agent 호출 권한은 별개로 봄
- 권한 문제는 bridge 가 해결하고, 호출 여부 판단은 `main` 이 해결함
- bridge `/app/run` 은 원래 사용자 프롬프트 앞에 현재 턴 전용 delegation authorization prefix 를 자동으로 붙임
- 이 prefix 는 다음을 명시함
  - 이번 턴에서 sub-agent, delegation, parallel work 사용을 사용자 요청으로 명시 허용
  - `main` 이 `agentplan` 의 `하드 트리거 + 점수제 + Route A/B/C` 흐름을 따르도록 지시
  - 작은 작업은 굳이 spawn 하지 말고, 큰 작업일 때만 자동 orchestration 하도록 지시
- 이렇게 해서 `spawn gate` 는 bridge prefix 로 열고, 실제 사용 여부는 `main` triage 결과에 맡김

## Route Definitions

- `Route A`
  - 작은 작업
  - `main` 이 직접 수정
- `Route B`
  - 경계 작업
  - `main` 이 직접 수정하되 필요하면 read-only 보조 역할만 호출
- `Route C`
  - 큰 작업
  - `main` 은 수정하지 않고 계약 고정, write set 분배, worker orchestration 만 수행

## Recommended Topology

- 기본 대형 작업 구성은 `main + worker_feature_1 + worker_feature_2 + worker_feature_3 + worker_shared + reviewer`
- `worker_feature_*` 는 각자 독립 기능 영역만 수정함
- `worker_shared` 는 공용 타입, 공용 유틸, 공용 컴포넌트, import 경로, 스키마, 이벤트 이름, 상수, shared test fixture 같은 공유 자산만 담당함
- 공유 자산이 거의 없으면 `worker_shared` 없이 feature worker 만 써도 됨
- 작업 규모가 작으면 `worker_feature_1` 하나만 쓰고 나머지는 띄우지 않음

## Worker Types

- `worker_feature_*`
  - 자기 기능 범위 파일만 수정
  - shared 자산 수정 금지
  - 다른 feature worker 의 파일 수정 금지
- `worker_shared`
  - shared 자산만 수정
  - feature 로직 확장 금지
  - feature worker 가 필요한 shared 변경 요청을 흡수하는 역할
- `reviewer`
  - read-only 검수만 수행
  - 수정 금지

## Worker Rules

- 각 `worker` 는 사전에 지정된 파일 집합만 수정함
- `worker` 끼리는 같은 파일을 절대 수정하지 않음
- 파일이 달라도 공유 계약을 임의로 바꾸지 않음
- 공용 타입, 스키마, 유틸, 이벤트 이름처럼 계약 중심 자산은 별도 담당을 지정하거나 `main` 이 선고정함
- feature worker 는 shared 파일을 수정하지 않음
- shared 파일 수정이 필요하면 `worker_shared` 에게만 할당함
- `worker_shared` 는 feature 파일을 수정하지 않음
- 각 worker 는 자기 write set 밖의 파일은 읽을 수 있어도 수정하지 않음

## Shared Asset Examples

- `types/*`
- `schemas/*`
- `shared/*`
- `components/common/*`
- 공용 훅, 공용 유틸, 공용 상수
- import path alias 설정
- 라우트 이름, 이벤트 이름, payload shape
- 테스트 fixture, mock contract, snapshot baseline

## Write Set Declaration

- `main` 은 worker 투입 전에 각 worker 의 write set 을 명시해야 함
- write set 은 파일 또는 디렉터리 단위로 적음
- shared 자산은 별도 `shared_set` 으로 분리해 선언함
- 선언이 애매하면 병렬화하지 않음

예시:

- `worker_feature_1 = [app/routes/palworld/*, app/features/palworld/hero/*]`
- `worker_feature_2 = [app/features/palworld/data/*, app/features/palworld/cards/*]`
- `worker_feature_3 = [tests/palworld/*, docs/palworld.md]`
- `worker_shared = [app/types/*, app/lib/utils/*, app/components/common/*]`

## Main Responsibilities

- 요구사항을 짧은 acceptance criteria 로 압축
- 공유 계약 고정
- 파일별 write set 분배
- 각 `worker` 에게 수정 범위와 금지 범위를 명시
- 병렬화가 안전한지 확인
- 결과 통합 판단
- `worker_shared` 가 필요한지 먼저 판단
- feature worker 와 shared worker 의 순서를 정함
- 통합 단계에서 누구에게 후속 수정이 필요한지 재할당

## Contract Freeze Checklist

- API request/response payload
- 상태 이름과 상태 전이
- 이벤트 이름과 발생 순서
- shared type 이름과 위치
- 공용 util 함수 시그니처
- 공용 컴포넌트 props
- 테스트 기대값의 기준값
- 각 worker 의 write set
- 수정 금지 경로

## Execution Order

1. `main` 이 acceptance criteria 와 contract freeze 를 고정
2. `main` 이 worker 별 write set 을 선언
3. shared 기반이 먼저 필요하면 `worker_shared` 가 선행 수정
4. feature worker 들이 병렬 수정
5. shared 정리나 import/path 수습이 남으면 `worker_shared` 가 후행 수정
6. `reviewer` 가 최종 read-only 검수
7. 수정이 필요하면 `main` 이 적절한 worker 에게 다시 할당
8. 모든 검수가 끝나면 `main` 이 종료 판단

## Reviewer Responsibilities

- 구현 후 read-only 로만 검수
- 계약 이탈 여부를 먼저 확인
- 회귀 위험, 빠진 검증, unsafe assumption 을 점검
- feature worker 가 shared 자산을 침범했는지 확인
- shared 수정이 feature 로직을 깨지 않았는지 확인
- import 경로, 타입 연결, 테스트 기대값이 맞는지 확인

## Parallelization Conditions

- write set 이 완전히 분리돼 있어야 함
- 공유 계약이 이미 고정돼 있어야 함
- 통합 시 충돌 가능성이 낮아야 함
- 위 조건이 애매하면 병렬화하지 않음
- shared 수정 책임자가 명확해야 함
- feature worker 가 shared 변경을 직접 처리하지 않도록 통제돼야 함

## Forbidden Patterns

- `main` 이 직접 코드 수정
- feature worker 둘 이상이 같은 shared 파일 수정
- feature worker 가 타입 파일, 공용 유틸, 공용 컴포넌트를 임의 수정
- shared worker 가 feature 로직까지 같이 수정
- contract freeze 없이 worker 부터 투입
- write set 선언 없이 병렬 수정 시작

## Expected Benefits

- `main` 컨텍스트를 구현 세부사항으로 덜 오염시킴
- `worker` 는 좁은 범위만 보고 빠르게 수정 가능
- 큰 작업에서 구현 속도와 집중도를 올릴 수 있음

## Main Risks

- 계약 고정이 애매하면 병렬 구현이 크게 꼬일 수 있음
- 통합 단계에서 타입, import, 테스트 기대값, 호출 순서 충돌이 날 수 있음
- `main` 이 직접 수정하지 않으므로 작은 통합 수정도 다시 `worker` 에게 넘겨야 할 수 있음
- `worker_shared` 에 변경이 몰리면 병목이 될 수 있음
- shared 자산 경계를 잘못 나누면 feature worker 가 자주 막힐 수 있음

## When This Model Fits

- 여러 파일군을 동시에 수정해야 하는 큰 기능 작업
- UI, 서버, 테스트, shared 자산이 모두 걸린 변경
- `main` 이 오케스트레이션에만 집중하는 편이 더 안전한 작업

## When This Model Does Not Fit

- 단일 파일 또는 소규모 수정
- shared 계약이 아직 흔들리는 초기 탐색 작업
- shared 자산 비중이 너무 커서 사실상 대부분이 한 worker 에게 몰리는 작업

## Operating Principle

- 작은 작업은 굳이 이 모델을 쓰지 않음
- 큰 작업에서만 `main planner-only + parallel workers + final reviewer` 구조를 사용함
