# Ouroboros Handoff Mode

이 문서는 우로보로스를 앱 안에서 직접 조종하는 대신, 별도 실행 엔진으로 두고 앱 스레드는 그 산출물만 읽어 이어받는 운영 방식을 정의한다.

## 핵심 원칙

- 우로보로스는 별도 엔진으로 실행한다.
- 앱 스레드는 우로보로스를 매 턴 직접 운전하지 않는다.
- 우로보로스가 남긴 계획서와 산출물을 읽고, 여기서 후속 작업을 이어간다.
- 즉:
  - 우로보로스 = 작업 수렴 엔진
  - 앱 스레드 = 작업 cockpit + 후속 실행/검토 공간

## 왜 이렇게 나누나

- 앱은 스레드 이동, 맥락 유지, 중간 정리, 병렬 대화가 편하다.
- 우로보로스는 `interview -> seed -> run -> evaluate` 루프가 강하다.
- 둘을 한 몸처럼 억지로 묶으면 제어 브릿지가 두꺼워지고, resume 실패 같은 런타임 문제가 작업 UX 전체를 망친다.
- 그래서 실행은 느슨하게 분리하고, 연속성은 계획서로 유지하는 쪽이 더 단순하다.

## 역할 분리

### 1. External Ouroboros

우로보로스가 담당하는 것:

- 자연어 목표 수신
- 인터뷰로 모호성 축소
- `seed` 계약 생성
- 필요 시 run/evaluate 수행
- 다음 단계에 필요한 계획서와 산출물 기록

우로보로스가 남겨야 하는 최소 결과:

- 현재 목표 요약
- 이번 루프의 판단
- 생성된 `seed` 경로 또는 관련 artifact 경로
- 다음에 사람이/앱이 해야 할 일
- 열려 있는 질문 또는 막힌 점

### 2. App Thread

앱 스레드가 담당하는 것:

- 우로보로스가 남긴 계획서 읽기
- 그 계획서 기준으로 코드 작업 이어가기
- 필요하면 추가 분석/검토/수정
- 최종 결과 정리

앱 스레드는 기본적으로:

- 계획서 소비자
- 후속 작업 실행자
- 결과 요약자

로 동작한다.

## 기본 흐름

### Step 1. 우로보로스 별도 실행

사용자는 우로보로스를 별도 CLI 세션에서 돌린다.

예:

- 목표 설명
- 인터뷰 진행
- `seed` 생성
- 개선 방향 확정

### Step 2. 계획서 남기기

우로보로스 세션이 끝나면, 반드시 읽기 쉬운 handoff note를 남긴다.

권장 위치:

- workspace root의 `OUROBOROS_PLAN.md`
- 또는 `docs/OUROBOROS_PLAN.md`

처음엔 Markdown이면 충분하다.

### Step 3. 앱 스레드가 계획서 읽기

여기 앱 스레드에서는 그 계획서를 읽고 아래를 바로 파악한다.

- 지금 목표가 뭐냐
- 이미 수렴된 계약이 뭐냐
- 다음 작업이 뭐냐
- 코드 작업을 어디서 시작해야 하냐
- 아직 답 안 난 질문이 뭐냐

### Step 4. 앱에서 후속 작업 진행

그 뒤 실제 코드 수정, 리뷰, 정리, 커밋 준비 같은 건 앱 스레드에서 이어간다.

즉 우로보로스는 계속 붙어서 운전하는 존재가 아니라:

- 앞단에서 문제를 수렴시키고
- 계획서를 남기고
- 앱이 그 바통을 받는 구조다.

## 계획서 최소 스키마

처음 버전은 아래 정도면 충분하다.

```md
# Ouroboros Plan

## Goal
- 이번 작업의 최종 목표

## Decided
- 인터뷰를 통해 확정된 내용

## Seed
- seed 경로 또는 계약 요약

## Next Actions
- 앱 스레드에서 바로 해야 할 일

## Open Questions
- 아직 안 풀린 질문

## Artifacts
- 관련 로그, seed, 결과 파일 경로
```

핵심은 예쁘게 쓰는 게 아니라, 앱 스레드가 이어받기 충분하게 쓰는 거다.

## 필수 연속성 항목

아래 중 하나라도 빠지면 handoff 품질이 급격히 떨어진다.

- 목표
- 확정된 계약
- 다음 액션
- 관련 artifact 경로

즉 최소 연속성 세트는:

- `goal`
- `decided`
- `next_actions`
- `artifacts`

다.

## 이 모드에서 하지 않을 것

- 앱이 매 프롬프트마다 우로보로스를 자동 호출
- wrapper/action 계층으로 우로보로스를 과하게 운전
- 우로보로스와 앱을 실시간 동기화된 단일 실행기로 취급

이 모드에선 계획서 handoff가 핵심이지, 완전한 실시간 일체화가 핵심이 아니다.

## 현재 추천 운영

지금은 이렇게 쓰는 걸 기본으로 둔다.

1. 우로보로스 CLI에서 목표를 수렴
2. 계획서와 seed/artifact를 남김
3. 앱 스레드에서 그 계획서를 읽음
4. 앱에서 실제 코드 작업과 검토를 이어감
5. 필요하면 다시 우로보로스로 다음 수렴 루프를 돈다

## 한 줄 결론

우로보로스를 매번 앱이 직접 조종하는 구조보다,
우로보로스가 계획서를 남기고 앱이 그 계획서를 읽어 이어가는 구조가
지금 단계에선 더 단순하고 덜 꼬인다.
